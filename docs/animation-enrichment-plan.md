# 戦国合戦 3D絵巻 リッチ化改善計画書

作成日: 2026-07-03
対象: サイト全体（トップ + 18合戦ページ + 共通エンジン）
実装担当: Codex（詳細手順は `docs/animation-implementation-plan-for-codex.md` 参照）

---

## 1. 診断: なぜ「チープ」に見えるのか

現状のコードを精査した結果、原因は9つに整理できる。重要度順。

### A. 3Dシーンの絵作り（最重要）

| # | 原因 | 該当箇所 | 何が起きているか |
|---|------|---------|----------------|
| 1 | **影・トーンマッピング・色空間が未設定** | `scripts/battle-sim.js` renderer初期化部 | `WebGLRenderer` がデフォルト設定のまま。sRGB出力もACESトーンマッピングも影もない。結果、全体が「のっぺり均一に明るい」CG初心者風の画になる。部隊が地面に接地して見えず浮いている |
| 2 | **空が単色** | `scene.background = 単色` + `FogExp2` | 背景に情報量ゼロ。時間経過も2色の単純lerpのみ。空・雲・太陽・地平線グラデがないので「箱庭感」が出る |
| 3 | **攻撃矢印がデバッグ表示そのまま** | `THREE.ArrowHelper` 使用 | ArrowHelperはThree.jsの開発用ヘルパー。均一な太さの線+円錐で、まさに「プログラマーアート」の象徴 |
| 4 | **兵士が均一な箱、旗が静止板** | `BoxGeometry(1.7,2.6,1.7)` の InstancedMesh、`PlaneGeometry` の旗 | 個体差なし、行進アニメは上下bobのみ、旗が一切なびかない。生命感がない |
| 5 | **戦闘しても何も起きない** | 状態遷移ロジックのみ | 「交戦」になってもラベル文字が変わるだけ。土煙・火花・銃火・炎・崩れの視覚表現が一切ない。合戦シミュレーターとして一番の見せ場が空白 |
| 6 | **地形がツルツル** | 頂点カラーのみの `MeshLambertMaterial` | 木も岩も家もない。スケール感の手がかりがなく、ミニチュア感すら出ない（ミニチュアなら逆にアリだが中途半端） |

### B. カメラ演出

| # | 原因 | 該当箇所 |
|---|------|---------|
| 7 | **カメラが単調** | `desiredCamera()` | autoモードは等速オービット(`az += dt*0.04`)のみ。「小早川裏切り」「鉄砲斉射」などの転機でもカメラは淡々と回り続ける。映像として山がない。揺れ・寄り・引きの演出ゼロ |

### C. Web UIのモーション

| # | 原因 | 該当箇所 |
|---|------|---------|
| 8 | **トップページがほぼ静止画** | `styles/index.css` | transitionは hover 160ms が2箇所だけ。スクロール登場アニメなし、ヒーローのKen Burnsなし、カードのマイクロインタラクションなし。「大河ドラマ風の重厚感」を謳うのに開幕演出がない |
| 9 | **合戦ページUIの遷移が雑** | `styles/battle-sim.css` | キャプションは単純fade、状態バッジは瞬間切替、タイムラインのイベント通過時に何も起きない。導入カットも静止画fadeのみ |

**総括**: 個々の要素は動いているが、「光・影」「戦闘の見せ場」「カメラの演出」「UIの所作」という映像的な4本柱がすべて素のままなので、データビジュアライゼーションのプロトタイプに見える。逆に言えば、**データ構造とエンジンの共通化は既にできている**ので、エンジン1本を磨けば18合戦が同時にリッチになる。ここが本サイト最大の強み。

## 2. 方針: どう考えてリッチにするか

### 設計原則（4つ）

1. **データ駆動を貫く** — 演出は `BATTLE_SCENE` 設定への追記（`effects`, `shot` 等）で表現し、エンジン側は汎用システムとして実装する。18ページに個別コードを書かない。これは将来のスマホアプリ版へのデータ移植性も担保する。
2. **No-build・r128継続** — script タグ + グローバル `THREE` の現行構成を崩さない。18ページが既に動いている構成を壊すリスクを取らない。examples モジュールが必要な場合のみ jsDelivr の `three@0.128.0/examples/js/` を追加 script タグで読む。
3. **品質ティア制** — 起動直後のフレームタイム計測で high / medium / low を自動判定し、影・パーティクル数・ポストエフェクトを段階的に落とす。`?fx=high|medium|low|0` で強制指定可。モバイル60fpsを死守する。
4. **`prefers-reduced-motion` 対応** — リッチにするほど酔い・アクセシビリティ配慮が必須。全モーションに reduce 時の代替（即時表示 or 減衰）を用意する。

### リッチ化の柱（効果の大きい順）

#### 柱1: ライティングと空 — 「同じデータで画が2段階良くなる」
- ACESトーンマッピング + sRGB出力 + ソフトシャドウ。これだけで全18合戦の画が一斉に締まる。
- 空をグラデーションドーム化（時刻で天頂・地平の2色が変化）+ 太陽ビルボード + 流れる雲スプライト。
- 夜戦（河越夜戦）・雨（桶狭間）・雪（大坂冬の陣）はドームの色データを差し替えるだけで対応。

#### 柱2: 戦闘エフェクト — 「見せ場を作る」
- 汎用パーティクルシステム（土煙 / 火花 / 銃火 / 炎+煙 / 破片）を1つ実装し、`BATTLE_SCENE.effects` 配列で時刻・座標・種類を指定。
- 例: 長篠は `volley`（鉄砲斉射の閃光列）、本能寺・大坂は `fire`、全合戦の交戦地点に `clash`（土煙+火花）。
- 「交戦」状態の部隊は自動で小規模な土煙を出す（データ追記ゼロでも全合戦が改善）。

#### 柱3: 部隊の生命感
- 旗のなびき（セグメント分割 + 正弦波変位）。
- 兵士の行進サイクル（インスタンスごとの位相ずれ + 進行方向への傾き）と個体色バラつき（`setColorAt`）。
- 移動中の隊列後方に土埃トレイル。壊滅時は散開 + 倒れ（スケールY潰し）。

#### 柱4: カメラ演出 — 「編集が入った映像にする」
- キャプション（転機）ごとにショット指定（`caption.shot = { type, az, polar, radius, shake }`）。寄り・引き・ドリー・クレーンの4種。
- 突撃・裏切り・総崩れの瞬間に減衰付きカメラシェイク。
- 指定がないキャプションは現行オービットにフォールバック（後方互換）。

#### 柱5: 矢印・注記の再設計
- `ArrowHelper` を廃止し、テーパー付きリボン + UVスクロールで「流れる」進軍矢印に置換。データスキーマ（`from/to/t0/t1/color`）は不変。
- 状態バッジのポップ切替をスケール+フェードのトゥイーンに。

#### 柱6: Web UIモーション
- トップ: IntersectionObserver によるカードの時差登場、ヒーロー画像の Ken Burns、カード hover 時の画像ズーム+金の光沢スイープ。
- 導入カット: 背景の緩ズーム + タイトルの字送り + 金罫線の伸長。
- タイムライン: イベント通過時にラベルがパルス、キャプションは刷毛はらい風ワイプ。

#### 柱7（任意・最後）: 環境ディテール
- 尾根に低ポリ樹木のインスタンス散布、城位置に簡易櫓。スケール感が生まれる。
- 効果は大きいがデータ調整が18合戦分必要なので最後。

### やらないこと
- Three.js のメジャーアップグレード（r160+ は ES Modules 必須で全ページ改修になる。得られる画質向上は上記施策で代替可能）
- WebGL ポストプロセッシング（EffectComposer）の初期導入 — ビネット・粒状感は CSS オーバーレイで再現し、リスクとモバイル負荷を回避。効果不足なら Phase 2 で検討
- 音声 — 効果は大きいが自動再生制約・素材権利の整理が必要。別企画とする
- 兵士の本格スケルタルアニメ — 俯瞰視点では費用対効果が低い

## 3. 追加する合戦の提案

### 何本あればいいか → **第二期6本を追加して計24本を推奨**

理由:
- 現在18本(第一期)。番組表メタファーに合わせ「第二期 六合戦」として区切りよく増やせる。
- 6本なら1本ごとに**新しい演出システムを1つずつ**担保でき、量産感が出ない（下表の「新演出」列）。
- 24本以降は追加の限界効用が下がる。25本目を作るより既存の演出密度を上げる方がサイトの印象に効く。SEO・回遊的にも24本で「コレクションサイト」として十分な厚み。

### 推奨6本（優先順）

REQUIREMENTS.md の評価軸（地形の意味 / 時間変化 / 視覚化しやすさ / 勝敗理由の明快さ / 演出パターンの新規性）で選定。既存18本と被る演出は避けた。

| 優先 | 合戦 | 年 | 対峙 | 表現テーマ | 新演出（エンジン拡張） |
|---|------|----|------|-----------|----------------------|
| 1 | **備中高松城の水攻め** | 1582 | 羽柴秀吉 vs 清水宗治 | 堤防築造→水位上昇→城の孤立。本能寺の報せで撤収 | **動的水位**（時刻で水面メッシュが上昇し地形を飲む）。唯一無二の画になる |
| 2 | **手取川の戦い** | 1577 | 上杉謙信 vs 柴田勝家 | 夜間の追撃、増水した川への殺到、織田軍の壊走 | **夜戦ライティング**＋川の渡河失敗演出（河越夜戦の夜スキームを発展） |
| 3 | **沖田畷の戦い** | 1584 | 島津家久 vs 龍造寺隆信 | 湿地の一本道、伏兵、釣り野伏せ、大将討死 | **地形拘束の隘路戦**（縦長フォーカスのカメラワーク）。九州編の拡充 |
| 4 | **摺上原の戦い** | 1589 | 伊達政宗 vs 蘆名義広 | 風向きの反転が勝敗を分けた野戦 | **風向きの可視化**（草原の波・砂塵の流向が時刻で反転）。東北編の入口 |
| 5 | **金ヶ崎の退き口** | 1570 | 織田軍(殿:木下秀吉) vs 朝倉・浅井 | 挟撃察知→全軍撤退→殿の防戦 | **「逃げ切る」逆算ドラマ**。追手との距離ゲージ的な緊張演出。勝ち負けでない物語型 |
| 6 | **三増峠の戦い** | 1569 | 武田信玄 vs 北条氏照・氏邦 | 山岳での大規模野戦、峠越えの機動 | **高低差カメラ**（クレーンショットが最も活きる立体戦） |

補欠（第三期候補）: 小田原征伐（包囲の規模感はあるが部隊移動が少なく現エンジン向きでない）、第二次上田合戦（城攻めシステムが必要）、岩屋城の戦い（玉砕戦の描き方に配慮が要る）。

### 追加順序の指針
リッチ化（実装計画書 M1〜M6）を**先に**完了させてから新合戦を作ること。新演出システム（動的水位・風向き）は M3 のエフェクト基盤の上に載せる設計のため、順序が逆だと二度手間になる。

## 4. スマホアプリ版への布石

本計画はアプリ化を見据えて設計してある。推奨パス:

1. **短期（このリッチ化と同時期）: PWA化** — `manifest.json` + Service Worker + アイコンを足すだけでホーム画面インストール・オフライン閲覧が可能に。工数最小で「アプリ」体験を先取り。
2. **中期: Capacitor でストア配布** — 現行サイトをそのまま WebView ラップ。3D は WebGL のまま動く。データ駆動化（原則1）と品質ティア（原則3）とタッチ操作（ピンチ/ドラッグは実装済み）が既に効いているため、追加工数は小さい。
3. **将来ネイティブ化する場合** — `BATTLE_SCENE` を純JSONに寄せておけば（現在はJSオブジェクト。ほぼJSON互換）、React Native + expo-gl 等への移植でも合戦データ資産をそのまま使える。

アプリ版で効く今回の決定: 演出のデータ駆動化 / 品質ティア自動判定 / reduced-motion対応 / タッチ前提のUI。**今回の実装計画に PWA 化を M8 として含めた**（軽いので）。

## 5. 実施順序と検収

実装は `docs/animation-implementation-plan-for-codex.md` の M0→M8 の順で行う。各マイルストーンは独立コミットとし、既存の検証モード（`?verify=1&t=<時刻>&weather=0`）でスクリーンショット比較検収する。

期待効果の目安:
- M1（光と空）完了時点で「チープ」の印象は概ね解消
- M3（戦闘エフェクト）完了時点で「見せ場のある映像」になる
- M6（UIモーション）完了時点でサイト全体の質感が揃う
